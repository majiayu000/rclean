# rclean v0.1.x Roadmap — 方案 C minus MCP

Status: Draft
Date: 2026-05-16
Package: `rclean-cli`
Binary: `rclean`
Version policy: 0.1.x only — every milestone bumps a patch (0.1.1 → 0.1.5)
Predecessor: `docs/specs/complete-spec.md`（v0.1.0，2026-05-06）

## 0. 摘要

v0.1.x 在 v0.1.0 的 trust model（ActionPlan + safe/caution/blocked + explain）之上做四件事：

1. **打地基** (0.1.1)：拆 `src/rules/mod.rs` 596 行单 match；合并 scan 阶段两次重复 walkdir；缓存 git 进程调用；引入 `tracing` 替换 `eprintln`。
2. **并发 + 配置** (0.1.2)：用 `ignore` crate 替换 `walkdir`，一举拿到并发遍历 + `.gitignore`-style 状态机；新增 `.rcleanignore` 用户配置文件。
3. **TUI + watch** (0.1.4)：基于 `ratatui` 的交互模式（替代纯文本编号选择）；基于 `notify` 的 watch 子命令做增量扫描。
4. **可恢复 + stamp** (0.1.3)：新增 graveyard 层（7 天可恢复，feature-gated）；新增 `rclean stamp` 跨生态打标-回收两阶段流程。

**不做 MCP server**。CLI 已能被 agent 通过 bash 调用，差异化在 trust model 而非协议层。

**不做向后兼容**。ActionPlan schemaVersion 1 → 2 是 hard break；v0.1.0 plan 文件会被拒绝并提示重新 scan。0.x 版本 semver 允许 patch 内 break。

## 1. Goals

- 保留 v0.1.0 全部已验证行为（scan / clean / explain / rules / ActionPlan / 三态安全分类）
- 并发扫描，至少 3× v0.1.0 在 monorepo 场景的吞吐
- 用户可写 `.rcleanignore` 排除路径，可选写 `.rclean.toml` 增加自定义清理规则
- 删除可恢复 7 天（graveyard mode），默认开启 OS 回收站
- TUI 交互模式做大量候选场景（>30 个候选）的过滤/搜索/选择
- watch 模式监听 lockfile 变化，增量更新候选清单
- 跨平台 CI（Linux / macOS / Windows）+ criterion benchmark suite

## 2. Non-Goals

- 不做 MCP server / 不做 daemon socket / 不做 RPC 协议
- 不做 cron 自动清理（watch 仅刷新候选，不自动执行清理）
- 不做 Docker / 系统级缓存清理
- 不做 GUI（TUI 不是 GUI）
- 不保留 v0.1.0 ActionPlan 兼容

## 4. 模块设计

### 4.1 rules/ 拆分（M1 必做，解 P0）

**当前问题**：`src/rules/mod.rs:20-354` 是一个 596 行 `classify_candidate` 巨型 match。

**新设计**：每个生态一个文件，导出 `pub fn classify(ctx: &ClassifyContext) -> Option<CandidateDraft>`。`rules/mod.rs` 用 enum + 顺序 dispatch：

```rust
// src/rules/mod.rs（≤120 行）

#[derive(Clone, Copy)]
pub enum Ecosystem {
    Node, Python, Rust, Jvm, Flutter, Dotnet, Ruby, Ios, Generic, User,
}

pub trait RuleSet {
    fn ecosystem(&self) -> Ecosystem;
    fn classify(&self, ctx: &ClassifyContext) -> Option<CandidateDraft>;
}

pub struct Classifier {
    builtin: [&'static dyn RuleSet; 9],
    user: Option<user::UserRuleSet>,
}

impl Classifier {
    pub fn classify(&self, ctx: &ClassifyContext) -> Option<CandidateDraft> {
        for rs in self.builtin.iter() {
            if let Some(draft) = rs.classify(ctx) { return Some(draft); }
        }
        self.user.as_ref().and_then(|u| u.classify(ctx))
    }
}
```

`ClassifyContext` 字段：
```rust
pub struct ClassifyContext<'a> {
    pub dir_name: &'a str,
    pub dir_path: &'a Path,
    pub parent: &'a Path,
    pub project_markers: &'a ProjectMarkers,   // package.json/Cargo.toml/...
    pub git_info: Option<&'a GitInfo>,
    pub mtime_secs: u64,
    pub size_hint: Option<u64>,
}

pub struct CandidateDraft {
    pub rule_id: &'static str,           // user rule 用 String? 见 §4.2
    pub category: Category,
    pub safety: Safety,
    pub why: String,
    pub restore_hint: Option<String>,
}
```

**关键决策**：内置 rule_id 用 `&'static str`；user rule 用 `Cow<'static, str>` 容纳运行期字符串。零额外分配在 hot loop 上。

### 4.2 user rules（.rclean.toml）

格式：

```toml
# .rclean.toml（项目根或 $XDG_CONFIG_HOME/rclean/rules.toml）

[[rule]]
id = "user.my_build"
name_glob = "my_build_*"           # 匹配目录名
parent_markers = ["Makefile"]      # 父目录必须有这些文件之一
category = "build"
safety = "safe"                    # safe / caution
why = "Custom Makefile build dir"
restore_hint = "make build"
```

**约束**：
- user rule 永远不能产出 `safety = "blocked"`（blocked 只能由 builtin 决定，防止用户配置反向降级安全模型）
- user rule 顺序在 builtin 之后（builtin 决定优先）
- user rule 文件解析失败时整个 user.rs 静默禁用并 `tracing::error!`（U-29：用户可见结果错误必须 error 级别，但单文件失败不阻止整体 scan）

### 4.3 .rcleanignore（glob 排除）

格式（标准 gitignore syntax，由 `ignore` crate 原生支持）：

```
# .rcleanignore
node_modules/some-protected/   # 单条排除
**/sacred/                     # 全局排除
!**/sacred/dist/               # 否定（保留 dist）
```

**实现**：直接交给 `ignore::WalkBuilder::add_custom_ignore_filename(".rcleanignore")`，复用 ripgrep 的 gitignore 状态机。零额外解析逻辑。

**层级合并**（最高优先级先）：
1. CLI flag `--ignore <glob>`
2. 项目根 `.rcleanignore`
3. 父目录链 `.rcleanignore`（git 同样行为）
4. 用户级 `$XDG_CONFIG_HOME/rclean/ignore`
5. 内置规则的 hard-skip 名单（`.git`, `.svn`, etc.）

### 4.4 scan/ 重构（M1 + M2）

**当前问题**：
- `src/scan.rs:85-150` 单线程 DFS
- `src/scan.rs:371` 和 `:225` 同一棵树扫两遍（dir_size + project_source_size）
- `src/scan.rs:457-480` git 子进程无缓存

**新设计**：

```rust
// src/scan/mod.rs
pub struct ProjectScanner {
    classifier: Classifier,
    git_cache: GitCache,
    options: ScanOptions,
}

impl ProjectScanner {
    pub fn scan(&self, roots: &[PathBuf]) -> Result<ScanReport, ScanError> {
        let walker = walker::build(&self.options);  // ignore::Walk + rcleanignore
        let visits: Vec<DirVisit> = walker
            .into_par_iter()   // rayon
            .filter_map(|entry| self.visit(entry))
            .collect();
        sizer::summarize(visits, &self.options)
    }
}
```

**关键点**：
- `walker::build` 返回 `ignore::WalkParallel`，自动并发遍历
- 每个目录访问一次（`visit`）就决定是否是 candidate；如果是 candidate 立刻在 ignore 的回调里启动 `sizer::dir_size` 子任务
- `sizer::summarize` 在一遍内同时输出 candidates、project 总源码大小、artifact 占比 —— 解决 §3.1 的二次遍历
- `GitCache` 内部 `RwLock<HashMap<PathBuf, GitInfo>>`，第一次访问某个 git root 时跑一次 `git status --porcelain -uall`，后续命中缓存

### 4.5 git_cache（M1）

```rust
// src/scan/git_cache.rs
pub struct GitCache(RwLock<HashMap<PathBuf, GitInfo>>);

pub struct GitInfo {
    pub is_repo: bool,
    pub git_root: Option<PathBuf>,
    pub dirty: bool,
}

impl GitCache {
    pub fn info_for(&self, path: &Path) -> GitInfo {
        // 1. 找最近的 .git ancestor（FS lookup，不调 git）
        // 2. 若该 root 已缓存，直接返回
        // 3. 否则跑一次 `git status --porcelain --untracked-files=normal`，记录 dirty 标志
        // 4. 跨 candidate 复用
    }
}
```

**禁止**：用 `libgit2` 引入大依赖。git 进程调用 + 缓存已足够（git 子进程在已 warm 的 git dir 上 <50ms）。

### 4.6 risk_score（M2）

新增字段 `Candidate.risk_score: f32`（0.0 = 完全安全，1.0 = 完全风险）。计算公式：

```
risk = w1·dirty_git + w2·recently_modified + w3·no_lockfile + w4·outside_known_root
```

权重（默认）：
- w1 = 0.40（dirty git worktree）
- w2 = 0.25（mtime < 7 天）
- w3 = 0.20（缺 `Cargo.lock` / `package-lock.json` / `Pipfile.lock` —— 重建可能失败）
- w4 = 0.15（路径接近 cwd 上层 / 跨文件系统）

**用途**：
- 输出层在 table 中加 risk 列
- ActionPlan v2 写入 `risk_score`，consumer 可基于此做策略
- TUI 用渐变色显示
- **不**改变 safe/caution/blocked 判定（trust model 不变，risk_score 是补充信号）

权重通过 CLI flag `--risk-weights "dirty=0.4,mtime=0.25,lockfile=0.2,boundary=0.15"` 覆盖；项目 `.rclean.toml` 可覆盖。

### 4.7 graveyard 层（M3）

**目录布局**：

```
$XDG_DATA_HOME/rclean/graveyard/
├── manifest.jsonl           # 一行一条记录（append-only）
└── 2026/05/16/
    └── 14h23m07-<sha>/
        ├── meta.json        # 完整 candidate + 原 path + plan_id
        └── payload/         # 实际被"删除"的目录内容（move 进来）
```

**记录格式**（`meta.json`）：

```json
{
  "id": "01HEXAGON-...",
  "original_path": "/Users/x/code/app/target",
  "size_bytes": 12884901888,
  "deleted_at": "2026-05-16T14:23:07Z",
  "plan_id": "01HEX-...",
  "rule_id": "rust.target",
  "risk_score": 0.12,
  "expires_at": "2026-05-23T14:23:07Z"
}
```

**子命令**：

```bash
rclean restore                      # 交互列出可恢复项
rclean restore --id <id>            # 按 id 还原
rclean restore --since 1h           # 还原最近 1 小时所有
rclean graveyard list               # 列出全部活跃 grave
rclean graveyard gc                 # 手动触发过期清理
```

**与 trash crate 关系**：
- 默认 `clean` 仍走 OS trash（trash crate）—— 系统级回收站，由 Finder/Files manager 管理
- `clean --graveyard` 走 rclean 自己的 graveyard —— 跨平台一致、可机器读、有 manifest
- 两者互斥；user 用 `--graveyard` 或 `--permanent` 二选一，否则走 trash 默认

**为什么不冗余**：trash crate 在 Linux 上行为不一致（XDG trash 是文件级而非"项目快照"），且无 manifest 让 user 知道"哪个 plan 删了什么"。rclean graveyard 是 plan-aware 的回收站。

**GC 策略**：
- 默认 7 天后过期，由下一次 `rclean scan` 或 `rclean graveyard gc` 触发清理
- 单 grave 超过 50 GB 时弹 warning，建议 `--permanent`
- graveyard 总大小通过环境变量 `RCLEAN_GRAVEYARD_QUOTA` 限制（默认无限）

**feature gate**：`graveyard` feature，默认开启。`cargo install rclean-cli --no-default-features` 完全编译掉 graveyard 代码。

### 4.8 watch 模式（M4）

**子命令**：

```bash
rclean watch <paths...> [--every 60s] [--write-plan auto.json]
```

**行为**：
1. 启动一次完整 scan，输出当前候选
2. 用 `notify::recommended_watcher` 监听以下文件：
   - `package-lock.json` / `pnpm-lock.yaml` / `yarn.lock`
   - `Cargo.lock`
   - `Pipfile.lock` / `poetry.lock` / `uv.lock`
   - `go.sum`
   - `.git/HEAD`（branch 切换）
3. 文件 mtime 变化触发**单项目**增量 rescan（只重扫触发的项目根，不重扫全部 roots）
4. 输出 diff：新增/消失/大小变化的候选
5. 不自动执行清理；若 `--write-plan` 指定，每次刷新写一份新 plan（带时间戳后缀）

**禁止**：
- watch 不自动 `clean`（守护进程清理违反 trust model）
- watch 不写入 graveyard
- watch 不长期占用 inotify 资源；20 分钟内无文件事件自动降级为 `--every` 轮询

### 4.9 stamp 模式（M3）

**子命令**：

```bash
rclean stamp <paths...>             # 打标当前 artifact 状态
rclean stamp <paths...> --sweep     # 找出未 touch 的，写 plan
```

**行为**：
- `stamp` 在每个 candidate 根下写 `.rclean-stamp`（隐藏文件），内容是 `{stamp_id, timestamp}`
- `--sweep` 扫描所有现存 candidate，找出 stamp 时间之后**未被修改**（mtime 没变过）的，写入 plan
- 灵感来自 cargo-sweep `--stamp/--file`，但 rclean 跨生态可用

**用途**：CI/CD 场景—— "build 之前打标，build 完成后回收没用上的 cache"。

### 4.10 TUI（M4，feature = "tui"）

**入口**：`rclean tui <paths...>` 或 `rclean clean --tui`

**界面布局**（ratatui）：

```
┌─ rclean tui ──────────────────────────────────────┐
│ Roots: ~/code   Reclaimable: 445 GB   Selected: 0 │
├───────────────────────────────────────────────────┤
│ [/]search  [space]toggle  [a]all-safe  [enter]plan│
├───────────────────────────────────────────────────┤
│ > [x] harness-workflow-runtime-phase2/target  101G│
│   [ ] my-app/node_modules                     2.4G│
│   [!] dirty/target                            512M│ (caution: dirty git)
│   [×] /usr/local/...                          ----│ (blocked: outside root)
├───────────────────────────────────────────────────┤
│ explain: rust.target — Cargo.toml exists, mtime=… │
└───────────────────────────────────────────────────┘
```

- `[x]` 选中、`[ ]` 未选、`[!]` caution（需手动）、`[×]` blocked（无法选）
- `/` 进入 fuzzy search 过滤
- `enter` 输出 plan 路径并退出
- `q` 取消退出
- 颜色：safe = 绿，caution = 黄，blocked = 灰，risk_score 高 = 橙色渐变

**降级**：terminal 不支持 alternate screen 时自动 fallback 到 v0.1.0 文本编号模式。

## 5. 数据模型变更（ActionPlan v2）

```json
{
  "schemaVersion": 2,
  "toolVersion": "0.1.5",
  "generatedAt": "2026-05-16T14:23:07Z",
  "deleteMode": "trash | graveyard | permanent",
  "roots": ["..."],
  "summary": { ... },
  "selected": [ /* candidate ids */ ],
  "projects": [ { ..., "candidates": [
      {
        "id": "01H...",                     // 新增：ULID，graveyard 关联
        "path": "...",
        "rule_id": "rust.target",
        "safety": "safe",
        "category": "build",
        "size_bytes": 12884901888,
        "risk_score": 0.12,                 // 新增

        "restore_hint": "cargo build",
```

**变更点**：
- `schemaVersion: 2` —— v1 plan 文件被显式拒绝（`PlanError::SchemaVersion { found: 1, supported: 2 }`），错误信息提示重新 `scan --write-plan`
- `candidate.id`（ULID）—— graveyard 关联、TUI 选中跟踪
- `candidate.risk_score`
- `deleteMode` 新值 `graveyard`

**Schema 文档**：`docs/specs/plan-schema-v2.json` 提供 JSON Schema 供机器消费。

### 6.3 全局 flag

- `--log <level>`：trace/debug/info/warn/error（替代 v0.1.0 的 --verbose；--verbose 保留为 alias = --log debug）
- `--config <path>`：覆盖默认 `.rclean.toml` 路径
- `--no-color`：禁用 ANSI 输出

## 7. 错误处理（U-17 / U-29）

**全局策略**：

- 公共错误类型用 `thiserror`：`ScanError` / `PlanError` / `CleanError` / `GraveyardError`
- 用户可见结果错误（缺 plan、blocked path 被强删等）→ `error!` 级别 + 进程非零退出
- 单文件单候选失败（permission denied、broken symlink）→ `warn!` 级别，candidate 标记 `unknown` 但 scan 继续（U-29 允许：用户最终看到的是"某些 candidate 标 unknown"，不是错误的最终答案）
- user `.rclean.toml` 解析失败 → `error!` 但 scan 继续（无 user rules，等价于 v0.1.0 行为）

**禁止**：
- `Result<T, String>` 全局替换为强类型错误（v0.1.0 全文件用 String，U-17 红线）
- `eprintln!` 替换为 `tracing::warn!`/`error!`，main.rs 安装 `tracing-subscriber`

## 12. 风险 / 回滚

| 风险 | 缓解 | 回滚预案 |
|------|------|---------|
| `ignore` crate 并发顺序非确定，破坏 plan id 稳定性 | 输出层强制按 path 字典序排序 candidates | M2 完成后跑 100× scan 比对结果哈希 |
| ratatui 在 Windows 终端兼容差 | 降级 fallback 到文本模式 + 显式提示 | TUI 是 feature flag，可临时禁用 |
| graveyard 磁盘占用失控 | `RCLEAN_GRAVEYARD_QUOTA` 上限 + 单 grave > 50GB warning | feature flag 可关闭 |
| watch 长时间运行 inotify 资源耗尽 | 20 分钟 idle 自动转轮询 | 用户随时 Ctrl-C |
| user rules 误把系统目录设 safe | user rule 不允许产出 `blocked`、且必须有 `parent_markers` | 解析阶段拒绝无 marker 的 user rule |

## 14. 决策日志

| 决策 | 候选 | 选择 | 理由 |
|------|------|------|------|
| 并发库 | jwalk / ignore / rayon+walkdir | **ignore** | 与 `.rcleanignore` 复用 gitignore 状态机；ripgrep 生态成熟 |
| MCP server | 做 / 不做 | **不做** | 用户决策：CLI 已能被 agent bash 调用；MCP 生态未成熟 |
| graveyard | 加 / 不加 | **加（默认 feature）** | trust model 闭环；plan-aware 优于 OS trash |
| rule 抽象 | trait obj / enum / TOML | **enum dispatch + user TOML** | 内置零开销 + 用户可扩展；gemini + grok 强共识 |
| 错误类型 | anyhow / thiserror | **thiserror** | 库代码，需要结构化错误而非黑盒上下文 |
| 日志 | tracing / log | **tracing** | 结构化字段、CLI 友好 EnvFilter |
| TUI 库 | ratatui / cursive | **ratatui** | 主流、活跃、跨平台稳定 |
| 时间库 | chrono / jiff | **chrono 暂留** | plan schema 已 chrono 序列化；切换收益小 |
