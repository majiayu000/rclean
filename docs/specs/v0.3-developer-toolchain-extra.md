# rclean v0.3 — Whole-machine developer cleanup

Status: Draft
Date: 2026-05-25
Package: `rclean-cli` (unchanged)
Binary: `rclean` (unchanged)
Version policy: 0.3.0 (minor bump; adds rules + extends `--home` expansion + new `--system` scope)
Predecessor: `docs/specs/v0.2-developer-mole.md`

## 0. 摘要

v0.2 给 rclean 划了一道边界：只清"开发者工具链产生的可重建杂物"。线上跑下来发现一个现实问题——**开发者的磁盘紧急情况里，最大头的几个目标恰恰被这条边界挡在外面**：

- macOS Sonoma+ 的动态壁纸缓存（`/Library/Application Support/com.apple.idleassetsd`）单台机器轻松 6–10 GB
- VSCode / Notion / 其他 Squirrel.Mac 应用的 `*.ShipIt` 旧更新包，每个 200 MB–1 GB
- Chrome 缓存动辄 1–2 GB
- GoogleUpdater 历史 700+ MB

v0.2 §2 Non-Goals 第一条把这些划在外面，理由是 "trust model 一致性 + 跨平台"。**v0.3 supersede 这条 Non-Goal**：我们把这些也纳入 rclean 的 trust model（safe/caution/blocked + ActionPlan + explain），并新增 `--system` 范围支持需要 `sudo` 的系统级路径。

**定位口号修订**：v0.1.x = "项目里的工件清理工具"；v0.2 = "开发者整机维护工具（用户级）"；**v0.3 = "开发者整机维护工具（含系统级 cache）"**。

**平台优先级**：macOS first（系统级规则全部 macOS-only，`#[cfg(target_os = "macos")]` gate）。Linux 继续保持现状（应用 cache 模式在 Linux 上路径完全不同，留 v0.4）。

## 1. Goals

- **In-scope 规则扩张**：从 16 条 → ≥ 23 条（详见 §3）
- **新增 `--system` flag**：扫描需要 `sudo` 的系统级路径（默认不开启，必须显式 `--system`）
- **扩展 `home_toolchain_paths()`**：覆盖 `~/.bun`、macOS `~/.cache`、`~/Library/Application Support/Google`（GoogleUpdater）
- **sudo 提示机制**：检测到系统路径的候选时，输出明确的 `sudo` 命令，**rclean 自身永不调 sudo**
- 保留 v0.2 全部 trust model（safe/caution/blocked + ActionPlan + symlink 防护 + dirty git 降级 + blocked 列表）
- `rclean doctor` 报告新规则的 applicable / skipped 状态
- `rclean rules` 列表自动包含新规则
- 跨平台 CI 继续保持绿

## 2. Non-Goals（v0.2 §2 修订版）

继承 v0.2 §2，但**修订第一条**：

- ~~❌ 不做系统级 cache 清理~~ **→ 改为：✅ 做系统级 cache 清理，但仅限**：
  - 路径在 `/Library/Application Support` 下、明确属于 macOS 系统服务的缓存（如 idleassetsd 动态壁纸）
  - 用户级 `~/Library/Caches/*` 下、属于已知 GUI 应用且**确定可重建**的缓存（ShipIt 更新包、Chrome cache、Lark cache 等）
  - 必须有 anchor name 精确匹配，不做泛 `~/Library/Caches/*` 一锅端

仍保留的 Non-Goals：
- ❌ **不**做 app 卸载器（plist / launch agent 反向工程超出 trust model）
- ❌ **不**做实时监控 / dashboard
- ❌ **不**做 GUI
- ❌ **不**做磁盘可视化
- ❌ **不**做 Windows 系统级规则（macOS first，Windows 留 v0.5）
- ❌ 核心清理命令不调系统服务、不改 `defaults write`
- ❌ **不**删除用户记录型数据（聊天记录、agent session、memory store；`~/.codex/sessions`、`~/.codex/memories`、`~/.claude/projects` 等保留在 blocked 列表）
- ❌ **不**自动调 `sudo`（仅输出推荐的 sudo 命令，由用户手动执行）

## 4. 架构改动：`--system` flag + sudo 提示机制

### 4.1 新 CLI flag

`src/cli.rs`：在 `scan` / `clean` 子命令加：

```
--system    Include system-level cache locations under `/Library/*`
            that require sudo for deletion. rclean will list these
            candidates and print the exact sudo command for each, but
            will NEVER invoke sudo automatically. Conflicts with `--yes`
            for system candidates (always requires manual confirmation).
```

### 4.2 路径展开

`home_toolchain_paths()` 保持不变（用户级路径）。新增 `system_paths()` 函数：

```rust
fn system_paths() -> Vec<PathBuf> {
    #[cfg(target_os = "macos")]
    {
        vec![PathBuf::from("/Library/Application Support")]
    }
    #[cfg(not(target_os = "macos"))]
    {
        Vec::new()
    }
}
```

`--home` 不展开 system paths；只有同时传 `--home --system` 或 `--system` 才会扫。

### 4.3 候选标记

`CandidateDraft` 加字段 `requires_sudo: bool`（默认 false）。系统级规则的分类器设为 true。

### 4.4 clean 行为

```
- 如果候选 requires_sudo=true：
  - dry-run 模式：正常显示，加 "⚠️  needs sudo" 标记
  - clean 模式：rclean 拒绝删除该候选，输出明确的 sudo 命令并 exit nonzero (除非 --yes 同时配 --print-sudo-only)
- ActionPlan 持久化 requires_sudo 字段
- replay (clean --plan) 时同样拒绝执行 sudo 候选，只打印命令
```

### 4.5 sudo 提示输出格式

```
The following candidates require root access. rclean will NOT run sudo
for you. Copy and run these commands manually:

  sudo rm -rf "/Library/Application Support/com.apple.idleassetsd"   # apple.idleassetsd, 6.9 GB

After running them, re-run rclean to verify the candidates are gone.
```

## 6. 安全保障

新增系统级路径风险更高，必须叠加防御：

| 保障 | 实现 |
|---|---|
| 受保护用户数据保留 | `~/.codex/sessions`、`~/.claude/projects` 等 blocked 列表保持不变（v0.3 不动） |
| Chrome cache vs Chrome data 不混淆 | rule anchor 强制 `Library/Caches/Google` 父路径，绝不匹配 `Library/Application Support/Google/Chrome` |
| symlink 防护 | 继续启用，系统路径上更严格（任何 symlink 候选直接 blocked） |
| `--system` 默认关闭 | 不显式传就完全不扫系统路径 |
| `--yes` 在系统候选上禁用 | 系统候选强制要求人工执行 sudo 命令，rclean 不代劳 |
| ActionPlan replay 时重新校验 | 即使旧 plan 包含系统候选，replay 仍会检查 requires_sudo + 路径未被劫持 |
| 不自动调 sudo | 即使 `--yes` 也只输出命令，不 `Command::new("sudo")` |

## 8. 兼容性

- **ActionPlan 向前兼容**：`requires_sudo` 字段 `#[serde(default)]`，旧 v0.2 plan.json 可被 v0.3 读取（缺字段视为 false）
- **ActionPlan 向后不兼容**：v0.3 生成的 plan 如果包含 system 候选，被 v0.2 binary 读取会忽略 `requires_sudo`，可能尝试无 sudo 删除导致权限错误。CHANGELOG 必须明确告知
- **rule_id 命名**：所有新 rule_id 与现有命名风格一致（`namespace.candidate`）

## 9. 风险与降级路径

| 风险 | 影响 | 降级 |
|---|---|---|
| Chrome cache rule 误匹配 Application Support 下的 Chrome（用户数据） | **高危**：可能删除书签/密码 | parent anchor 严格区分 `Library/Caches/Google` vs `Library/Application Support/Google`；加单元测试断言；如果发现误匹配立即在 `apply_path_safety` 加专项 blocked |
| `apple.idleassetsd` 路径在未来 macOS 改变 | rule 失效（扫不到），但不会误删 | 监控 macOS release notes |
| Squirrel.Mac 应用未来不再用 `*.ShipIt` 命名 | 旧应用仍命中，新应用扫不到 | rule 失效=安全，最多浪费一次扫描 |
| 用户在自定义 launchd 配置里把 idleassetsd 数据迁到了别处 | 扫不到 | rule 走 anchor 严格匹配，不会误删别处 |
| 系统级 cache 删除后 macOS 系统服务异常 | 中度风险（多数情况下系统会自动重建） | restore_hint 明确告知；safety 不降级到 caution（因为可逆），但 CHANGELOG 加显眼提示 "first-run will redownload aerials" |

## 11. 决策记录

- **为什么 v0.3 supersede v0.2 §2 第一条而不是出 v1.0**：架构变化不大（多加 1 个 flag + 几条规则），现有 trust model 完全可以延伸到新规则。语义上仍然是"可重建缓存"，没有偏离 rclean 的核心定位。
- **为什么不自动调 sudo**：rclean 的 trust model 核心是"用户审计每一步"。自动 sudo 会引入 askpass 机制，复杂度爆炸，且违反 "user is the final decision maker" 原则。打印命令让用户复制粘贴是最稳的妥协。
- **为什么 macOS first**：v0.2 已经是 macOS first。Linux 下 GUI 应用 cache 路径完全不同（XDG）且 Linux 用户少有"图形登录会话内存满"的痛点。Linux 系统级规则放 v0.4 评估。
- **为什么 Puppeteer 还是不加**：Puppeteer 在 npm postinstall 隐式下载，已经被 `node.node_modules` 规则间接覆盖（删 node_modules 会一起删）；独立全局缓存罕见。如果用户反馈需要，v0.4 加。
