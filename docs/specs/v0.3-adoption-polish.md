# rclean v0.3 — Adoption Verification and Power-User Polish

Status: Draft
Date: 2026-07-04
Package: `rclean-cli`
Binary: `rclean`
Predecessor: `docs/specs/v0.2-best-ux.md` (2026-07-02, fully shipped as
v0.2.0 except the one carry-over noted below)

## 0. Summary

v0.2.0 shipped the full distribution pipeline (tag-triggered releases,
binstall metadata, Homebrew tap with automated formula bumps), the
one-command default experience (TUI by default, no-arg flow, inline
explain, recovery summaries), and goal-oriented cleanup (`rclean free`,
staleness ranking). The v0.2.0 draft release carries all 13 verified
artifacts; publishing it and running `cargo publish` are human steps.

v0.3 closes the loop on three fronts:

1. **Prove the install paths stay working.** Distribution now has three
   entry points (crates.io build, binstall archive, brew formula) that
   can each silently rot. A smoke workflow exercises all three against
   the live release so breakage is a red run, not a user report.
2. **Finish and deepen the interactive surface.** The v0.2 spec's
   `free` design included handing the proposal to the interactive
   selector pre-selected; only the plan-write path shipped. Large
   candidate sets also need sorting and filtering beyond fuzzy search.
3. **Keep the paper surface honest.** README status, release notes,
   and benchmarks should track reality automatically, not by memory.

Boundaries are unchanged: no general-purpose cleaning, no sudo, no
background deletion (README "Product Boundaries").

## 1. Carry-over from v0.2

**CO1. `free --interactive`.** v0.2 §3.3 C1 specified the proposal
could be "handed to the interactive selector with the set
pre-selected"; v0.2.0 shipped only the ActionPlan path. Add
`rclean free <size> --interactive` (TTY required): the selector opens
with the proposed set pre-selected, the user adjusts, and the flow
continues through the normal confirm + recoverable-delete pipeline.
Same hard limits as plan mode: only safe non-sudo candidates are
pre-selected, and deselection/reselection cannot cross into blocked or
report-only rows (the selector already enforces this).

## 2. Workstream E — Install-path verification (P1)

**E1. Post-release smoke workflow.** A `workflow_dispatch` +
weekly-scheduled workflow that, on macOS and Linux runners:
- `cargo binstall rclean-cli --no-confirm` → `rclean --version`
- `brew install majiayu000/rclean/rclean` → `rclean --version` (macOS)
- `cargo install rclean-cli --locked` → `rclean --version`
Each path is a separate job so one failure doesn't hide another. Runs
red until the v0.2.0 release is published and the crate exists on
crates.io — that first green run is the release-verification evidence
the checklist asks for.

**E2. Release notes from the changelog.** The release workflow's draft
step extracts the tag's `## X.Y.Z` section from `CHANGELOG.md` and uses
it as the release body, so the human publish step is review-and-click
instead of copy-paste.

## 3. Workstream F — Interactive depth (P2)

**F1. Selector sorting and filtering.** In the TUI selector:
- `s` cycles sort: size desc (default) → staleness desc → risk asc
- `c` cycles category filter: all → deps → build → cache → test
- Header shows the active sort/filter; `/` fuzzy search composes with
  both. Selection state survives re-sorting and filtering.

**F2. Empty-result guidance.** When a scan finds zero candidates in an
interactive context (no-arg flow, `tui`, `clean` without `--json`),
print one hint line: try `rclean scan --home` for toolchain caches or
`rclean scan --tmp` for temp worktrees. Machine outputs (`--json`,
exit codes) are unchanged.

## 4. Workstream G — Honest paper surface (P2)

**G1. README refresh for the v0.2 surface.** "Current Status" and
Usage still describe the pre-0.2 CLI. Document: no-arg default flow,
TUI default with `?` explain, `free <size>`, staleness column and
`stale_after_days`, recovery summaries, `completions`/`man`, and the
three install paths. One pass, kept consistent with `--help` output.

**G2. Benchmark trend job.** CI job (main pushes only) runs the
existing criterion `scan_throughput` bench and uploads the report as
an artifact, so scan-performance regressions are visible per-commit.
No gating — trend visibility first, thresholds only if drift shows up.

## 5. Non-Goals

- No new cleaning rules in this milestone (rule additions continue via
  the normal per-rule issue flow with safety review).
- No watch/stamp expansion; they stay as shipped.
- No auto-publish of releases or crates — human gates stay.
- No TUI mouse support, panes, or theming.

## 6. Sequencing

E1/E2 and G1 land any time (E1 turns green only after the human
publish steps). CO1 before F1 (both touch the selector; CO1 is the
smaller diff). F2 and G2 are independent.
