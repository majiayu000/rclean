# rclean v0.2 — Developer-grade mole

Status: Draft
Date: 2026-05-19
Package: `rclean-cli` (unchanged)
Binary: `rclean` (unchanged)
Version policy: 0.2.x; every milestone bumps a patch
Predecessor: `docs/specs/v0.1.x-roadmap.md`

## 0. 摘要

v0.2 把 rclean 的范围从 "项目工件" 扩展到 "**所有由开发工具产生的可重建杂物**"。对手不再是 kondo/npkill，而是 [tw93/mole](https://github.com/tw93/mole) 的 `mo purge` 子命令在开发者市场的份额。

我们**不**做 mole 的系统级清理（用户 Library cache、浏览器、app 卸载、磁盘可视化、实时监控）—— 那是消费者市场，需要 macOS 特定 hack，会破坏 rclean 的跨平台和 trust model 一致性。

我们**做**的是：把 `.gradle` / `~/.cargo/registry/cache` / Xcode DerivedData / iOS Simulators / Docker dangling / IDE workspace cache 这一类东西也纳入 rclean 的 `safe/caution/blocked` 信任模型 + ActionPlan + `explain` 流程。

**定位口号**：v0.1.x 是 "项目里的工件清理工具"；v0.2 是 "**开发者的整机维护工具**"。

**平台优先级**：macOS first。macOS-only 的规则（DerivedData、Simulators、Homebrew cache）用 `#[cfg(target_os = "macos")]` gate；Linux/Windows CI 继续保持绿，规则数较少。

## 1. Goals

- 保留 v0.1.x 全部 trust model（safe/caution/blocked + ActionPlan + explain + graveyard）。
- 扩展规则覆盖范围：**至少 8 类新规则**（详见 §3）。
- 新增 `~` 模式：`rclean scan --home`（或 `rclean scan ~`）走开发者整机扫描预设，不再要求显式给路径。
- 新增 TUI 选择器（`--tui` flag，沿用现有 ActionPlan 模型，候选 >20 个时默认开启）。
- Homebrew tap 分发：`brew install majiayu000/tap/rclean`。
- 文档：每类规则有"为什么删它安全 / 删之后怎么恢复"的一行说明（沿用 `restore_hint` 字段）。

## 2. Non-Goals

- ❌ **不**做系统级 cache 清理（用户 `~/Library/Caches/*`、`/Library/Caches/*`、浏览器 cache、Spotify/Slack cache、Trash 等）。这是 mole 的市场。
- ❌ **不**做 app 卸载器。删 `.app` 必然涉及 plist / launch agent / preferences 反向工程，超出 trust model 能保证的范围。
- ❌ **不**做实时监控 / CPU/GPU dashboard。
- ❌ **不**做 GUI。TUI 不是 GUI。
- ❌ **不**做磁盘可视化（DaisyDisk 风格）。`scan --json` 已经能驱动外部可视化。
- ❌ **不**做 Windows / Linux 特定的规则（registry cleanup、systemd journal、apt cache 等）。但所有现有规则跨平台保持。
- ❌ 核心清理命令（`scan` / `clean` / `doctor`）**不**重启系统服务 / **不**改任何 `defaults write` 之类的系统状态。rclean 的清理模型只删可重建文件，不调系统旋钮。若未来存在 `agent optimize` 这类 app 设置辅助，必须独立于清理模型、默认 dry-run，并且需要显式 `--yes` 才能写入。

## 3. 规则范围（新增）

按价值×安全性排序。**每一行都要在 v0.2.x 落地前完成 trust model 评估**。

### 3.1 Phase 1：高价值、高确定性（v0.2.0 必做）

| 规则 ID | 路径 | safety | 平台 | 重建方式 |
|---|---|---|---|---|
| `xcode.derived_data` | `~/Library/Developer/Xcode/DerivedData` | safe | macOS only | 下次 Xcode build 时重建 |
| `xcode.simulators_unused` | `~/Library/Developer/CoreSimulator/Devices/<UUID>` (未使用 + 非默认) | caution | macOS only | `xcrun simctl create` 重建 |
| `gradle.caches` | `~/.gradle/caches/<old-versions>` | caution | 跨平台 | 下次 build 时重下 |
| `maven.local_repo_stale` | `~/.m2/repository/<old>` (访问时间 > 90 天) | caution | 跨平台 | 下次 build 时重下 |
| `cargo.registry_cache` | `~/.cargo/registry/cache` + `~/.cargo/registry/src` | safe | 跨平台 | `cargo build` 时重下 |
| `cargo.git_db` | `~/.cargo/git/db` (整子目录) | safe | 跨平台 | `cargo build` 时重 clone |
| `pip.cache` | `~/Library/Caches/pip` / `~/.cache/pip` | safe | 跨平台 | 重 `pip install` |
| `node.global_store` | `~/.npm/_cacache` / `~/.pnpm-store` / `~/Library/Caches/Yarn` | safe | 跨平台 | 包管理器重下 |

**Phase 1 投产门槛**：每条规则要有 (a) 真实测试 fixture 验证 marker 检测、(b) 对应的 `explain` 输出包含 `restore_hint`、(c) README 表格更新、(d) `--json` schema 不变。

### 3.2 Phase 2：要小心做（v0.2.1）

| 规则 ID | 路径 | safety | 备注 |
|---|---|---|---|
| `docker.dangling_images` | Docker API → 列 dangling images | caution | 不删 named/tagged，需要 docker daemon 运行 |
| `docker.unused_volumes` | Docker API → 列无引用 volumes | **blocked-by-default** | volume 里可能是用户数据，require `--include-blocked` 显式选 |
| `docker.build_cache` | Docker API → `builder prune` 等价 | caution | 走 docker SDK，不走 shell |
| `vscode.workspace_storage_orphan` | `~/Library/Application Support/Code/User/workspaceStorage/<hash>` | caution | hash 对应的 workspace 已不存在才删 |
| `jetbrains.system_caches` | `~/Library/Caches/JetBrains/<IDE>/<version>/<index>` | caution | 旧版本 IDE 的 index |

Docker 类规则需要新增 optional `docker` feature flag 和 docker daemon 连接逻辑。**没装 docker / docker daemon 没启动 = 静默跳过**（不是 error，因为不影响其他规则）。

### 3.3 Phase 3：可选、低优先级（v0.2.2+）

- `homebrew.cleanup`：等价于 `brew cleanup --prune=all`。考虑直接 shell out 到 brew 而非重实现。
- `android.sdk_unused_versions`：旧版本 build-tools / system-images / NDK。
- `flutter.pub_cache`：`~/.pub-cache`。

## 4. 新子命令 / Flag

### 4.1 `rclean scan --home`

等价于：`rclean scan ~` + 启用所有 `target_os` 匹配的全局规则 + 自动 caution 模式（因为 `~` 是 broad-root，正常要 `--allow-broad-root`，`--home` 隐式带）。

**安全约束**：
- `--home` 强制 `--dry-run` 第一次跑（即第一次跑 scan 完后必须 `--write-plan` 再 clean，不能 inline 走 `--all`）
- `--home` 永远跑 graveyard 模式不允许 `--permanent`
- `--home` 必然有 interactive confirm

### 4.2 `rclean scan --tui`

候选 > 20 时默认开启。基于 `ratatui` 写一个最小 TUI：
- 列表显示候选 + 大小 + safety + ruleId
- 空格选择/取消，回车确认
- `e` 在 candidate 上按下 = `explain` overlay
- `/` 搜索过滤
- 退出导出 ActionPlan，复用现有 `clean --plan` 流程

**不**做：实时进度条、列表外的额外面板、自定义主题、动画。

### 4.3 `rclean doctor`

新增辅助命令（可选）：检测环境（macOS 版本 / Xcode 是否装 / Docker 是否运行 / Homebrew 路径），输出每条 v0.2 规则的 "applicable / skipped / why"。Agent 友好。

## 5. 架构影响

### 5.1 规则结构扩展

现有 `src/rules/<ecosystem>.rs` 模式继续用。Phase 1 加 5 个新文件：
```
src/rules/xcode.rs      # macOS-only via cfg
src/rules/gradle.rs     # 全局 ~/.gradle 规则（与现有 jvm.rs 项目级规则共存）
src/rules/maven.rs
src/rules/cargo_global.rs
src/rules/pip.rs
src/rules/node_global.rs
```

**和 v0.1.x 项目级规则的区别**：项目级规则需要 `parent_markers`（`Cargo.toml`、`package.json`）；全局规则不需要，因为路径本身就是 marker（`~/.cargo/registry/cache` 路径名就唯一）。

新增 trait method `Rule::is_global_path() -> bool`，walker 看到全局路径时不要求 parent marker。

### 5.2 Walker 影响

`ignore::WalkParallel` 引擎不变。新增的"扫 home 目录预设"在 `scan` 入口处展开成多个 root path（`~/.cargo` / `~/.gradle` / `~/Library/...`），各自走一次 walker。每个 root 用现有的 `IgnoreMatcher::build` 继续支持 `.rcleanignore`。

### 5.3 `--json` schema

**不**改 schemaVersion。新规则只是新的 `ruleId` 字符串；现有消费者代码无须更新。

### 5.4 Docker feature

`docker` 走 cargo feature flag，默认关闭：

```toml
[features]
default = ["graveyard"]
docker = ["bollard"]   # bollard 是 docker rust client
```

Docker daemon 不在线时 `scan` 不报错，只是 docker 类规则不产出候选。`explain` 一个 docker 路径在 docker 不可用时返回 exit code 3 (unknown)。

## 6. 安装与分发

### 6.1 Homebrew tap

新建 `majiayu000/homebrew-tap`，每次 release tag 自动更新 formula。CI：在 release workflow 后跑一个 `gh workflow run` 更新 tap 仓库。

`brew install majiayu000/tap/rclean` → 装 `rclean` CLI。

### 6.2 cargo-binstall

确保 `cargo binstall rclean-cli` 工作（需要 release artifact 命名规则配合）。

### 6.3 不做 GUI

mole 的 GUI 卖 $9 lifetime。我们不进这个市场——GUI 是另一个工程，违反"只做必要功能"。

## 7. 里程碑

| 版本 | 内容 | 工程量估计 |
|---|---|---|
| 0.2.0 | Phase 1 八条规则 + `--tui` + `--home` + Homebrew tap | 4-6 周 |
| 0.2.1 | Phase 2 (Docker / IDE caches) + `doctor` 命令 | 2-3 周 |
| 0.2.2 | Phase 3 + 用户反馈打补丁 | 2 周 |

**Ship gate**：v0.2.0 在 macOS 上能扫整个 `~` 在 < 5 秒内完成（M4 Pro 基准），单次释放 ≥ 5 GB 在"50 个项目 + 老 Xcode 安装"基线机器上。

## 8. 风险

| 风险 | 缓解 |
|---|---|
| 误删 Xcode `Cryptex` (mole 文档明确点出过) | Phase 1 必须包含一个 "in-use vs unused" 检测；不确定就归 caution |
| `~/.m2` / `~/.gradle` 删了导致大重下耗带宽 | 默认走 graveyard，restore_hint 写明 |
| Docker daemon 连接卡死 | bollard 调用必须有 5 秒 timeout |
| `--home` 误清用户文件 | trust model 不变：runtime/system 路径全 blocked；graveyard 默认开启 |
| macOS-only 规则破坏跨平台测试 | 所有 macOS 规则 `#[cfg(target_os = "macos")]` gate；CI 在 Linux/Windows 继续跑剩余规则 |
| TUI 增加二进制体积 | `ratatui` 走 optional feature `tui`，默认开启但可关掉 |

## 9. 不做向后兼容

- v0.1.x → v0.2.0 时，`schemaVersion` 保持 `1`（增量兼容）。
- 现有 CLI flag 全部保留。
- 新 ruleId 是新增的字符串值；不重命名旧 ruleId。
- 唯一的 break：`--home` 是新 flag，没有旧行为可破坏。

## 10. 不在本 SPEC 范围

- watch/daemon 模式（v0.1.x SPEC 列为 M6，延后）
- MCP server 集成（明确决定不做，原因见 v0.1.x SPEC）
- 任何"自动定时清理"功能（cron / launchd 集成）—— 与 trust model "每次都要人审"冲突
