# rclean v0.2 — Distribution and First-Minute Experience

Status: Draft
Date: 2026-07-02
Package: `rclean-cli`
Binary: `rclean`
Predecessor: `docs/specs/v0.1.x-roadmap.md` (方案 C minus MCP, 2026-05-16)

## 0. Summary

v0.1.x delivered the engineering foundation: parallel scan, three-state
safety classification, `explain`, ActionPlan v2 with deletion-time
revalidation, graveyard recovery, an optional ratatui selector, and
broad rule coverage (project artifacts + global toolchain caches).

The gap between rclean and "the best developer disk-clean tool" is no
longer features or safety. It is **distribution** and the **first
minute of use**:

- The only install path is `cargo install --path .` from a checkout.
  There is no crates.io release, no prebuilt binary, no Homebrew
  formula, and no release workflow in `.github/workflows/`.
- The default flow is two commands (`scan`, then `clean`) with a
  numbered text selector. The TUI exists but is feature-gated off
  (`Cargo.toml`: `default = ["graveyard"]`).
- The trust model (explain, ActionPlan, graveyard) is the product's
  differentiator, but a first-time user never sees it unless they read
  the docs.

v0.2 makes three commitments:

1. **Zero-friction install** — publish to crates.io, ship prebuilt
   binaries with `cargo-binstall` support, and provide a Homebrew tap.
2. **One-command default experience** — `rclean` with no arguments
   scans the current directory and drops into the interactive selector;
   the TUI becomes a default feature with the existing text fallback;
   explain and the graveyard safety net become visible inside the flow.
3. **Goal-oriented cleanup** — `rclean free <size>` computes the
   smallest safe set that reclaims the requested space, ranked by
   staleness and risk, and routes it through the existing ActionPlan
   review model.

North-star metric: **from "heard about rclean" to "first GB freed,
recoverably" in under two minutes.**

## 1. Goals

- `cargo install rclean-cli`, `cargo binstall rclean-cli`, and
  `brew install <tap>/rclean` all work.
- GitHub Releases carry signed/checksummed prebuilt binaries for
  macOS (arm64, x86_64), Linux (x86_64, aarch64), and Windows (x86_64).
- `rclean` with no subcommand = scan cwd + interactive selection +
  confirm + recoverable delete, in one invocation.
- TUI compiled in by default; automatic fallback to the existing text
  selector when the terminal lacks alternate-screen support (fallback
  path already exists in `src/tui/mod.rs`).
- Inline explain inside the selector: pressing `?` on a candidate shows
  rule id, safety reasoning, and project markers without leaving the UI.
- Every recoverable delete ends with an explicit restore hint
  ("freed 12.3 GB — recoverable for 7 days via `rclean restore --id <id>`").
- `rclean free <size>` produces a reviewable ActionPlan that meets the
  target using only `safe` candidates, preferring stale ones.
- Staleness surfaced in ranking and output ("untouched 94d"), reusing
  the existing activity/git data collected during scan.

## 2. Non-Goals

Unchanged from README "Product Boundaries" and the v0.1.x roadmap:

- No general-purpose privacy cleaner, app uninstaller, or malware scanner.
- No GUI (the TUI is not a GUI). No MCP server, daemon, or RPC layer.
- No automatic background deletion; `watch` refreshes candidates only.
- No `sudo` escalation; system-scope items stay report-only.
- No new deletion semantics: everything destructive still flows through
  the scan → classify → (plan/select) → revalidate → delete pipeline.
- `rclean free` never selects `caution`, `blocked`, or `report-only`
  candidates, and never bypasses interactive/plan confirmation.

## 3. Workstreams

### 3.1 Workstream A — Distribution (P0)

**A1. Release workflow with prebuilt binaries.**
Add a tag-triggered GitHub Actions release workflow (cargo-dist or a
hand-written matrix) that builds the five target triples, attaches
archives + SHA-256 checksums to the GitHub Release, and embeds
`cargo-binstall`-compatible metadata in `Cargo.toml` (`[package.metadata.binstall]`
if the default URL template does not match).

Acceptance:
- Pushing a `v*` tag produces a draft release with 5 artifacts + checksums.
- `cargo binstall rclean-cli` resolves and installs from that release.
- CI green on the release workflow itself (dry-run job on PRs).

**A2. crates.io publication.**
One-time `cargo publish` for `rclean-cli`, plus a documented release
checklist (`docs/release/`): version bump policy, `cargo publish --dry-run`
gate, tag → release workflow → crates.io ordering, and README install
section update (remove "After public release" phrasing once done).

Acceptance:
- `cargo install rclean-cli` works from a clean machine.
- README Install section shows crates.io as the primary path.

**A3. Homebrew tap.**
Create `majiayu000/homebrew-rclean` with a formula pointing at the
release artifacts (bottle-from-binary style), and a small automation
step in the release workflow that bumps the formula on new tags.

Acceptance: `brew install majiayu000/rclean/rclean && rclean --version`.

### 3.2 Workstream B — Default one-command experience (P0)

**B1. TUI as a default feature.**
`default = ["graveyard", "tui"]`. Binary-size and dependency cost
(crossterm + ratatui) is acceptable for a CLI whose primary surface is
interactive selection. Keep `--no-default-features` builds working; the
text selector remains the compiled fallback everywhere.

**B2. No-arg default flow.**
`rclean` with no subcommand behaves as: scan cwd → interactive
selection → confirmation summary → recoverable delete (graveyard by
default, trash where graveyard is disabled). Equivalent to a new
implicit `clean --interactive` path; all existing subcommands are
untouched. Non-TTY invocation without a subcommand prints help and
exits (agents and scripts must keep using explicit subcommands —
no-arg never deletes non-interactively).

**B3. Inline explain in the selector.**
`?` on a highlighted candidate opens a pane rendering the same content
as `rclean explain <path>`: rule id, safety state and reasoning, project
markers, size, staleness. Data is already in `Candidate`/`ProjectReport`;
this is a TUI rendering task, not a new data path.

**B4. Visible safety net after delete.**
After any recoverable clean (graveyard or trash), print one line:
freed bytes, recovery mechanism, expiry window, and the exact restore
command. Permanent deletes print freed bytes and an explicit
"not recoverable" note.

### 3.3 Workstream C — Goal-oriented cleanup (P1)

**C1. `rclean free <size>`.**
New subcommand: scan (respecting the usual scan flags), then select the
smallest set of `safe` candidates whose total size ≥ target, ranked by
`(staleness, risk_score asc, size desc)`. Output: the proposed set with
per-candidate reasoning, written as an ActionPlan (`--write-plan`
default on) or handed to the interactive selector with the set
pre-selected. If the target cannot be met from safe candidates, say so
explicitly and show the gap — never widen selection silently (U-29).

**C2. Staleness-aware ranking everywhere.**
Compute "days since last meaningful activity" per candidate from the
existing project activity data, show it as a column in scan table, TUI,
and JSON (`staleness_days`), and use it in the "Biggest wins" ordering.
Threshold config (`stale_after_days`, default 30) lives in `.rclean.toml`.

### 3.4 Workstream D — Polish (P2)

**D1. Shell completions + man page.** `rclean completions <shell>` via
`clap_complete`, man page via `clap_mangen`, both wired into the release
artifacts and the Homebrew formula.

**D2. Scan progress feedback.** During scans longer than ~1s, stream a
single-line progress indicator (projects visited / candidates found) to
stderr; silent under `--json` unless `--verbose`.

## 4. Sequencing

1. A1 → A2 → A3 (A3 depends on release artifacts from A1).
2. B1 → B2 → B3/B4 (B2 depends on the TUI being default).
3. C2 → C1 (free-mode ranking consumes staleness).
4. D1/D2 anytime after A1.

Version plan: A + B ship as `0.2.0`; C as `0.2.x` minors per repo
semver policy; D rides along.

## 5. Risks

- **Windows targets are the least-tested surface.** The release matrix
  makes Windows a first-class artifact; CI already runs three OSes, but
  trash/graveyard behavior on Windows needs a smoke test before A1 ships.
- **TUI-by-default changes the default binary footprint.** Mitigation:
  `--no-default-features` stays supported and documented.
- **`free` mode is the first "rclean chooses for you" surface.** It must
  stay inside the safe class and inside the review pipeline; the spec
  treats any silent widening of selection as a release blocker.
