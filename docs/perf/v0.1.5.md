# rclean v0.1.5 — Performance Report

Status: Skeleton. Numbers filled in when M5 (#26) wraps the benchmark
sub-task and runs on the same hardware that captured the v0.1.0
baseline.

## Methodology

All measurements taken on the maintainer's reference machine:

- Apple Silicon, 16 cores, 128 GB RAM, APPLE SSD AP8192Z (NVMe)
- macOS 15.x
- Rust `stable` toolchain matching the release tag

Each row is a hyperfine run with `--warmup 1 --runs 5`, scanning a
real workspace (path documented per-row). The same workspace is
scanned with every revision; cache state is warmed by the warmup pass.

The in-repo `cargo bench --bench scan_throughput` runs against a
synthetic 50-project fixture and is useful for catching regressions
on small inputs but is **not** the source of truth for these numbers —
process startup dominates at that size. Real-workspace hyperfine wins.

## Reproducer

```bash
# Pick a real workspace. Examples used in v0.1.0 audit:
#   ~/Desktop/code/work  (30 GB / 238k files / 9 projects)
#   ~/code               (smaller, faster signal for development)
WORKSPACE="${1:-$HOME/Desktop/code/work}"

# Warm the page cache once, then sample 5.
cargo build --release
hyperfine --warmup 1 --runs 5 \
  "./target/release/rclean scan $WORKSPACE --json --min-size 0 > /tmp/rclean-out.json"
```

For the M5 final report we also capture `time` output (user / sys /
wall) per run to distinguish IO-bound from CPU-bound regressions —
the v0.1.0 baseline showed ~89% sys time, so a sys-time regression
matters more than a user-time one.

## SPEC target

`docs/specs/v0.1.x-roadmap.md` §8 sets the M5 release bar at
**≥ 3× v0.1.0 baseline** on a monorepo-shaped fixture. PR-2d
(`ignore::WalkParallel + rayon`, not yet open) is the most likely
single change to clear that bar; the four M1 PRs together are
expected to land somewhere around 1.5–1.8× on a single-repo workspace
and 2× on a multi-repo monorepo.

The M5 ship gate is binary: either this table shows ≥ 3× on the
documented reference workspace, or 0.1.5 doesn't tag.

## Caveats

- The reference workspace mtimes change as I work in it. Numbers
  drift slightly run to run for that reason — σ in the table
  captures that.
- macOS APFS caches aggressively. The "real cold cache" number can
  only be approximated (boot the machine, scan once). The published
  numbers are warm-cache mean over 5 runs, which is the user-
  perceived steady state, not the worst case.
- These numbers are not portable to Linux ext4 / btrfs / xfs. A
  Linux-specific row is M5 follow-up work; reach out via #41's CI
  matrix output for raw Ubuntu-runner timings.
